# Database Guidelines

> Database patterns and conventions for this project.

---

## Overview

This project **does not use a traditional database**. Instead, it uses **file-based JSON persistence** for configuration and authentication data. This is a deliberate architectural choice for simplicity and portability.

---

## Data Storage

### Configuration Files

| File                         | Purpose                           | Format                 |
| ---------------------------- | --------------------------------- | ---------------------- |
| `configs/models.json`        | Model definitions (Gemini models) | JSON array             |
| `configs/auth/auth-N.json`   | Browser context auth data         | Playwright JSON format |
| `.env` or `.env.development` | Environment variables             | KEY=VALUE format       |

### Authentication Data

**Location**: `configs/auth/auth-N.json` (N = 0, 1, 2, ...)

**Format**: Playwright browser context state

```json
{
    "cookies": [...],
    "origins": [...],
    "localStorage": [...]
}
```

**Generated by**:

- `npm run setup-auth` - Interactive auth setup (launches browser)
- VNC login in Docker

## Configuration Loading

### ConfigLoader Pattern

**File**: `src/utils/ConfigLoader.js`

**Pattern**: Load from environment variables first, then override with JSON files

```javascript
// src/utils/ConfigLoader.js:17-33
loadConfiguration() {
    const config = {
        apiKeys: [],
        httpPort: 7861,
        streamingMode: "fake",
        round: "round",
        sessionErrorThreshold: 3,
        maxContexts: 1,
        logLevel: "INFO"
    };

    // Override with environment variables
    if (process.env.PORT) {
        config.httpPort = parseInt(process.env.PORT, 10);
    }
    if (process.env.API_KEYS) {
        config.apiKeys = process.env.API_KEYS.split(",");
    }
    if (process.env.LOG_LEVEL) {
        config.logLevel = process.env.LOG_LEVEL.toUpperCase();
    }

    return config;
}
```

### Model Loading

**File**: `src/utils/ConfigLoader.js:110-133`

```javascript
loadModelList() {
    const modelsPath = path.join(process.cwd(), "configs", "models.json");

    if (fs.existsSync(modelsPath)) {
        try {
            const modelsData = JSON.parse(fs.readFileSync(modelsPath, "utf-8"));
            this.modelList = modelsData.models;
            this.logger.info(`Loaded ${this.modelList.length} models from config`);
        } catch (error) {
            this.logger.warn("Failed to load models.json, using defaults:", error.message);
            this.modelList = this._getDefaultModels();
        }
    } else {
        this.modelList = this._getDefaultModels();
    }
}
```

## Data Access Patterns

### Read-Only Configuration

Most configuration is read-only at runtime:

```javascript
// Load once at startup
const config = ConfigLoader.loadConfiguration();
const models = config.modelList;

// Access throughout application
if (models.includes(requestedModel)) {
  // ... process request
}
```

### Auth Data Loading

**File**: `src/auth/AuthSource.js`

**Pattern**: Load auth files dynamically, validate and deduplicate

```javascript
async loadAuthData() {
    const authDir = path.join(process.cwd(), "configs", "auth");
    const files = fs.readdirSync(authDir)
        .filter(file => file.startsWith("auth-") && file.endsWith(".json"));

    const authData = [];
    for (const file of files) {
        const data = JSON.parse(fs.readFileSync(path.join(authDir, file)));
        // Validate and deduplicate by email
        if (!authData.some(a => a.email === data.email)) {
            authData.push(data);
        }
    }

    return authData;
}
```

## File Structure Guidelines

### When to Create New Config Files

1. **New model definitions** → Edit `configs/models.json`
2. **New authentication accounts** → Add `configs/auth/auth-N.json`
3. **New environment variables** → Add to `.env.example` and `ConfigLoader.js`

### File Naming

- **Config files**: lowercase with hyphens
  - Example: `models.json`, `auth-0.json`
- **Environment files**: `.env`, `.env.development`, `.env.production`

---

## Migrations

**No migration system** - configuration files are version-controlled.

**To update config**:

1. Edit the JSON file directly
2. Restart the server
3. Changes are loaded on startup

---

## Data Validation

### JSON Schema Validation

**No schema validation** - files are trusted

**Error handling**:

```javascript
try {
  const data = JSON.parse(fs.readFileSync(filePath, "utf-8"));
  return data;
} catch (error) {
  this.logger.warn(`Failed to load ${filePath}:`, error.message);
  return defaultValue;
}
```

## Best Practices

### DO

- ✅ Use `ConfigLoader` for all configuration access
- ✅ Validate auth data when loading (deduplicate, check fields)
- ✅ Log configuration on startup for debugging
- ✅ Provide sensible defaults when files are missing
- ✅ Use environment variables for runtime configuration

### DON'T

- ❌ Read config files directly - use `ConfigLoader`
- ❌ Modify config files at runtime
- ❌ Store sensitive data in `models.json` (use `.env` for secrets)
- ❌ Assume config files always exist - handle missing files gracefully

## Common Mistakes

### 1. Reading Config Files Directly

**Bad**:

```javascript
const models = JSON.parse(fs.readFileSync("configs/models.json"));
```

**Good**:

```javascript
const configLoader = new ConfigLoader();
const models = configLoader.loadModelList();
```

### 2. Hardcoding Paths

**Bad**:

```javascript
const authDir = "configs/auth";
```

**Good**:

```javascript
const authDir = path.join(process.cwd(), "configs", "auth");
```

### 3. No Error Handling

**Bad**:

```javascript
const data = JSON.parse(fs.readFileSync(filePath));
```

**Good**:

```javascript
try {
  const data = JSON.parse(fs.readFileSync(filePath, "utf-8"));
  return data;
} catch (error) {
  this.logger.warn(`Failed to load ${filePath}:`, error.message);
  return defaultValue;
}
```
